🔴 Live Streaming STT

      
A deep-dive into Deepgram's real-time transcription mode — what it is, how it works, and how the system design would change.

    


    
    

      
What We Built vs What Live Streaming Is


      

        

          The current POST /stt endpoint uses Deepgram's
          pre-recorded mode: you upload a complete audio file,
          wait for it to be fully processed, and receive the entire transcript in one response.
          This is perfect for processing existing recordings.
        

        

          Deepgram's live streaming mode works completely differently:
          instead of sending a finished file, you open a persistent two-way connection and send
          audio in small chunks as it is being recorded — in real time. Deepgram sends back partial
          transcripts within milliseconds as it hears each word, before you've even finished speaking the sentence.
        

      


      

        
          

            Dimension
            Pre-recorded (current)
            Live Streaming
          

        
        
          

            Protocol
            HTTP POST — one request, one response
            WebSocket — persistent two-way connection that stays open
          

          

            Audio input
            Complete file uploaded all at once
            Raw audio chunks sent continuously, 20–250ms at a time
          

          

            When you get results
            After the entire file is processed (seconds to minutes)
            Within ~300ms of each word being spoken — mid-sentence
          

          

            Result type
            One final, complete transcript
            A stream of interim results (partial) followed by final results per utterance
          

          

            Use case
            Podcast transcription, call recording analysis, meeting notes
            Live captioning, voice commands, real-time translation, voice-driven UI
          

          

            Connection lifecycle
            Opens → sends file → closes
            Opens → streams audio chunks → receives words → stays open → closes when done
          

          

            Latency
            Equal to audio duration + processing time
            ~300ms from spoken word to transcript (near real-time)
          

        
      

    


    
    

      
How Deepgram Live Streaming Works — Step by Step

      

        

          

            

            
1. Client opens a WebSocket connection

            

              Instead of a POST request, the client connects to
              wss://api.deepgram.com/v1/listen
              with model parameters in the URL query string (model, language, diarize, etc.).
              The connection stays open for the entire duration of the session.
            

          

          

            

            
2. Client streams raw audio bytes

            

              The microphone (or audio source) is read in small binary chunks — typically 20 to 250 milliseconds of audio at a time.
              Each chunk is sent over the WebSocket as a binary message the moment it is captured.
              No buffering, no waiting for the sentence to finish.
            

          

          

            

            
3. Deepgram sends back interim transcripts

            

              After receiving each chunk, Deepgram responds with a JSON message containing
              is_final: false — a best-guess transcript of what it heard so far.
              These are fast but may change as more context arrives. You can use these to show
              "live captions" that update as the person speaks.
            

          

          

            

            
4. Deepgram sends final utterances

            

              When Deepgram detects a pause (end of sentence or thought), it sends a message with
              is_final: true — the committed, corrected transcript for that utterance.
              This is what you save to your database or display as confirmed captions.
            

          

          

            

            
5. Client sends a CloseStream message

            

              When the user stops speaking or the session ends, the client sends a special
              {"type":"CloseStream"}
              JSON message. Deepgram flushes any remaining audio, sends the last final transcript,
              and closes the connection cleanly.
            

          

        


        

          💡
          A WebSocket is different from HTTP. HTTP is like sending a letter and waiting for a reply — one exchange and done. A WebSocket is like a phone call — the line stays open and both sides can send messages to each other at any time, in any order, for as long as they want.
        

      

    


    
    

      
What Would Change in the System Design


      

        
❌ Remove — no longer needed

        
The file upload (multipart/form-data): There is no file to upload. Audio arrives as a stream of small binary packets, not as a complete file. The HTTP POST endpoint and UploadFile handler would be replaced entirely.

      


      

        
✅ Add — new infrastructure required

        
WebSocket endpoint in FastAPI: FastAPI supports WebSockets natively with @app.websocket("/stt/live"). The handler would accept the client's WebSocket connection, open a second WebSocket to Deepgram, and act as a proxy — forwarding audio chunks from the client to Deepgram and forwarding transcript messages from Deepgram back to the client, simultaneously.

      


      

        
✅ Add — concurrent I/O handling

        
Async bidirectional message routing: The server needs to listen on two WebSocket connections at the same time — one from the browser and one to Deepgram. This requires two concurrent async tasks running in parallel using asyncio.gather() or asyncio.create_task(). If either connection drops, both must be closed cleanly.

      


      

        
🔄 Modify — different database tracking

        
Database logging strategy: With pre-recorded, you log one row per request. With live streaming, a session can last minutes. The tracking approach changes: you either log one row per final utterance received (many small rows per session), or accumulate the full session transcript in memory and write one row when the session ends. The latency field changes meaning too — it becomes the time from a spoken word to receiving the final transcript for that utterance, not end-to-end request time.

      


      

        
🔄 Modify — authentication approach

        
API Key in WebSocket: HTTP headers cannot be set on WebSocket connections from a browser. The X-API-Key header would instead be passed as a URL query parameter when opening the connection (wss://host/stt/live?api_key=...), or in the first message sent after the connection opens. The server validates it immediately and closes the connection if it is missing or wrong.

      


      

        
🔄 Modify — frontend interaction

        
Browser microphone access: The frontend would use the MediaRecorder API to access the microphone, capture audio in chunks, and send each chunk over a WebSocket. The transcript results would appear word-by-word on screen in real time — the UI would update continuously rather than waiting for a single final result to display.

      

    


    
    

      

        
Summary — Pre-recorded vs Live at a Glance

        

          

            
Pre-recorded (implemented)

            
    
              
  • Simple HTTP POST — easy to build and test
    • 
              
  • Works on any audio file regardless of when it was recorded
    • 
              
  • One response contains the complete, final transcript
    • 
              
  • Latency = full audio duration + processing time
    • 
              
  • Best for: archives, recordings, uploads
    • 
            

          

          

            
Live Streaming (reflection)

            
    
              
  • WebSocket — more complex, requires bidirectional handling
    • 
              
  • Only works with live microphone or real-time audio source
    • 
              
  • Words appear on screen as they are spoken (~300ms delay)
    • 
              
  • Latency = milliseconds per word regardless of session length
    • 
              
  • Best for: captions, voice UI, real-time translation
    • 
            

          

        

        

          🎯
          Both modes use the same Deepgram Nova-2 model and produce the same quality transcript. The difference is entirely about when you need the results — after the fact (pre-recorded) or as it happens (live streaming).

Endpoint 1 — Speech-to-Text

      

        

          POST
          /stt
          Powered by Deepgram Nova-2
        

        

          
What it does: You upload an audio file (MP3 or WAV) and this endpoint sends it to Deepgram's Nova-2 speech recognition model. Nova-2 listens to the audio, converts every spoken word into text, figures out what language is being spoken, measures how long the audio is, and identifies which words were said by which speaker when multiple people are talking.


          
When would you use it? Any time you have recorded audio that you need in text form — meeting recordings, voice memos, call center conversations, podcast transcripts, or voice commands in an app.

What you send

              
    
                
  • audio — the file itself (MP3 or WAV, up to 25 MB). Sent as a form attachment, not JSON.
    • 
                
  • X-API-Key — your API key in the header.
    • 
              

            

            

              
What you get back

              
    
                
  • transcript — the full spoken text as a single string.
    • 
                
  • language — auto-detected language code (e.g. es for Spanish, en for English).
    • 
                
  • duration — length of the audio in seconds.
    • 
                
  • speakers — list of segments, each with speaker ID, start time, end time, and what they said.
    • 
              

            

          


          
{
  "transcript": "Hola, ¿cómo estás hoy?",
  "language": "es",
  "duration": 3.42,
  "speakers": [
    { "speaker": 0, "start": 0.0, "end": 3.42, "text": "Hola, ¿cómo estás hoy?" }
  ]
}


          

            💡
            Deepgram charges per second of audio processed. The duration field in every response tells you exactly how many seconds were billed. This is tracked automatically in the database under external_usage.

Endpoint 3 — Intelligent Document Processing

      

        

          POST
          /idp/analyze
          Powered by OpenAI GPT-4o Vision
        

        

          
What it does: You upload a photo or scan of a document (ID card, passport, invoice, receipt, driver's license — anything with text on it). This endpoint sends the image to OpenAI's GPT-4o model, which looks at it like a human would, identifies what type of document it is, reads every visible field, and returns all the information structured as clean JSON. It also draws colored boxes on the image to show exactly where it found each piece of information.


          
When would you use it? Automating data entry from scanned documents, verifying identity documents in onboarding flows, extracting invoice data for accounting systems, digitizing paper records, or any scenario where humans currently read documents and type the information into a system.

The exact token counts are tracked in external_usage on every request.

What it does: Returns the full history of every API call made through this system, pulled directly from the PostgreSQL database. ☁️ Architecture Point 2

      
Diseño conceptual de arquitectura cloud — Caso A: Análisis masivo de llamadas en call center.

    


    
    

      
    


    
    

      
1. Descripción del Caso de Uso y Retos Técnicos

      

        

          Un call center de gran escala puede manejar entre 50,000 y 500,000 llamadas diarias.
          Cada llamada genera un archivo de audio que, en el modelo tradicional, requiere revisión humana para extraer insights:
          ¿el agente fue empático? ¿el cliente quedó satisfecho? ¿de qué tema trató la llamada?
          El objetivo de esta arquitectura es procesar ese volumen de manera completamente automática,
          en la nube, produciendo transcripciones, análisis de sentimiento, detección de temas y reportes por agente y por campaña,
          sin intervención humana en el pipeline.
        

        
RETOS TÉCNICOS PRINCIPALES

        

          

            
📦 Volumen e ingesta

            
    
              
  • Cientos de miles de archivos de audio diarios llegando de forma continua
    • 
              
  • Distintos formatos (WAV, MP3, OGG) y distintas duraciones (2–45 min)
    • 
              
  • Picos de carga en horarios de mayor actividad del call center
    • 
            

          

          

            
⚡ Latencia y tiempo real

            
    
              
  • Los supervisores necesitan alertas de llamadas críticas en <5 minutos
    • 
              
  • Transcripción y análisis deben terminar antes del siguiente turno
    • 
              
  • Soporte para modo live (streaming) para monitoreo en tiempo real
    • 
            

          

          

            
🌍 Multilingüismo

            
    
              
  • Call centers atienden en múltiples idiomas según región
    • 
              
  • El modelo STT debe detectar idioma automáticamente
    • 
              
  • Los análisis de sentimiento deben ser culturalmente precisos
    • 
            

          

          

            
🔒 Privacidad y compliance

            
    
              
  • Datos de clientes altamente sensibles (PII, datos financieros)
    • 
              
  • Regulaciones como GDPR, PCI-DSS según país
    • 
              
  • Trazabilidad y auditoría completa de cada procesamiento
    •

3. Flujo de Datos — Entrada, Procesamiento y Salida

      

        

          

            

            
① Entrada — Grabación llega al sistema

            

              El PBX o sistema de grabación del call center sube el archivo WAV/MP3 a Azure Blob Storage (RAW)
              vía API REST o SFTP. Cada archivo va etiquetado con metadatos: ID de agente, ID de campaña, timestamp de inicio,
              número de cliente (anonimizado para compliance). En el momento del upload, Azure Event Grid
              dispara un evento que crea un mensaje en Azure Service Bus sin ninguna acción manual.
            

          

          

            

            
② Procesamiento STT — Transcripción con Deepgram

            

              Azure Container Apps consume el mensaje de la cola, descarga el audio de Blob Storage
              y lo envía a Deepgram Nova-2. El modelo devuelve la transcripción completa con
              diarización (quién habló en qué momento), idioma detectado y marcas de tiempo por palabra.
              Para llamadas largas (>30 min), el audio se divide en segmentos de 5 minutos para paralelizar
              el procesamiento y reducir la latencia total.
            

          

          

            

            
③ Análisis NLU — Azure OpenAI GPT-4o

            

              La transcripción es enviada a Azure OpenAI (GPT-4o) con un prompt estructurado que extrae:
              sentimiento general (positivo / neutro / negativo / escalar 0–100),
              temas detectados (billing, soporte técnico, cancelación, queja, etc.),
              entidades clave (productos mencionados, números de cuenta — con enmascaramiento PII),
              resumen ejecutivo de la llamada (<3 oraciones),
              y score de calidad del agente (adherencia a protocolo, empatía, resolución).
              Los resultados se estructuran como JSON para escritura directa en base de datos.
            

          

          

            

            
④ Persistencia — Múltiples destinos en paralelo

            

              El orquestador escribe de forma paralela en tres destinos:
              Azure PostgreSQL recibe la fila de metadatos y scores (escritura rápida, consultable con SQL estándar).
              Cosmos DB recibe el documento JSON completo con la transcripción y el resumen.
              ADLS Gen2 archiva todos los datos crudos y procesados para compliance y re-entrenamiento futuro.
            

          

          

            

            
⑤ Salida — Reportes y alertas

            

              Azure Synapse Analytics corre queries programadas (cada hora o bajo demanda) que agregan los datos
              de PostgreSQL y el Data Lake para producir métricas consolidadas.
              Power BI se conecta a Synapse y presenta dashboards en tiempo cuasi-real: ranking de agentes
              por score de calidad, distribución de sentimiento por campaña, temas más frecuentes, evolución temporal.
              Si el análisis detecta sentimiento muy negativo durante el procesamiento, Azure Monitor
              dispara una alerta al supervisor en menos de 5 minutos tras la finalización de la llamada.
            

          

        

      

    


    
    

      
4. Justificación de los Servicios Cloud Elegidos

      

        

          
            

              Servicio
              Por qué este servicio
              Por qué no otra alternativa
            

          
          
            

              Deepgram Nova-2
              Mejor WER (Word Error Rate) en inglés y español para audio de call center con ruido. Diarización nativa, detección automática de idioma y marcas de tiempo por palabra en una sola llamada API.
              Azure Speech-to-Text tiene menor precisión en acentos regionales. OpenAI Whisper no ofrece diarización nativa ni streaming en tiempo real.
            

            

              Azure OpenAI GPT-4o
              Modelo líder en comprensión de texto complejo. Permite análisis multi-tarea en un solo prompt (sentimiento + temas + entidades + resumen), reduciendo el número de llamadas API.
              Modelos más pequeños (GPT-3.5) son menos precisos en análisis de matices de sentimiento. Modelos open-source requieren infraestructura propia y ajuste fino costoso.
            

            

              Azure Blob Storage
              Almacenamiento de objetos de bajo costo con 11 nueves de durabilidad. Integración nativa con Event Grid para triggers automáticos sin polling.
              ADLS Gen2 es superior para analytics pero más costoso como landing zone. Blob Storage es óptimo para la ingesta inicial de archivos binarios.
            

            

              Azure Service Bus
              Garantiza entrega al menos una vez, reintentos automáticos, dead-letter queue para mensajes fallidos y desacoplamiento total entre ingesta y procesamiento.
              Azure Event Hubs está orientado a streaming de alta frecuencia y no garantiza orden ni exactly-once delivery. Para jobs discretos de procesamiento de archivos, Service Bus es más apropiado.
            

            

              Azure Container Apps
              Escala automáticamente a 0 cuando no hay mensajes en la cola (KEDA) y a cientos de réplicas en momentos pico — sin gestionar infraestructura de Kubernetes manualmente.
              Azure Functions tiene límite de ejecución de 10 min (máx 60), insuficiente para llamadas largas. AKS requiere gestión activa del cluster. Container Apps es el punto medio ideal.
            

            

              Azure PostgreSQL Flexible
              Base de datos relacional gestionada con alta disponibilidad, backups automáticos y soporte nativo para JSON. Compatible con el ORM (SQLAlchemy) ya usado en este proyecto.
              Cosmos DB no es óptimo para queries SQL relacionales con JOINs y GROUP BY que producen los reportes de agentes. PostgreSQL es mejor para datos estructurados y métricas.
            

            

              Azure Cosmos DB
              Latencia de lectura <10ms para servir transcripciones completas bajo demanda. Modelo de datos flexible (JSON) ideal para documentos de longitud variable sin esquema fijo.
              PostgreSQL no es eficiente almacenando textos de transcripción de 5,000+ palabras como campos JSONB si se consultan frecuentemente por fragmento de texto.
            

            

              Azure Synapse Analytics
              Permite consultar el Data Lake (ADLS) con SQL serverless sin mover datos. Escala elásticamente para queries históricas pesadas sin costo fijo de infraestructura.
              Azure SQL Warehouse (dedicado) tiene costo fijo aunque no se use. Synapse serverless paga por TB leído — ideal para analytics episódicos sobre histórico grande.
            

            

              Power BI
              Integración nativa con Azure Synapse y PostgreSQL. Dashboards interactivos sin necesidad de código. Familiar para supervisores y equipos de gestión no técnicos.
              Grafana está más orientado a observabilidad de infraestructura. Tableau requiere licencias más costosas. Power BI Premium está incluido en muchas licencias M365 empresariales.
            

          
        

      

    


    
    

      
5. Escalabilidad, Disponibilidad y Seguridad

      

        

          
📈 Escalabilidad

          
    
            
  • Azure Container Apps + KEDA: escala de 0 a N réplicas basándose en el número de mensajes en Service Bus. 0 costo cuando no hay llamadas.
    • 
            
  • Service Bus Premium con particionamiento: distribuye la carga entre múltiples brokers para alta frecuencia de mensajes.
    • 
            
  • Deepgram concurrencia: soporta miles de transcripciones simultáneas — se paga por minuto de audio, no por instancia.
    • 
            
  • Azure OpenAI rate limits: gestionados con retry + backoff exponencial. Batch API para análisis no urgentes a menor costo.
    • 
            
  • Synapse serverless: escala automáticamente para queries de reportes, sin provisionar capacidad fija.
    • 
            
  • Medallion architecture: Bronze (raw) → Silver (cleaned) → Gold (aggregated). Permite re-procesar solo lo que cambió sin rehacer todo el pipeline.
    • 
          

        

        

          
🔄 Disponibilidad

          
    
            
  • Azure SLA 99.9%+ en todos los servicios gestionados (Blob, PostgreSQL, Cosmos DB, Service Bus).
    • 
            
  • Geo-replication: Blob Storage y Cosmos DB replicados en zona secundaria con failover automático.
    • 
            
  • PostgreSQL Flexible con High Availability: réplica standby en otra zona con failover automático en <60 segundos.
    • 
            
  • Service Bus Dead-Letter Queue: mensajes que fallan 3 veces van a DLQ para revisión, sin pérdida de datos.
    • 
            
  • Container Apps health checks: reinicio automático de instancias no saludables.
    • 
            
  • Circuit breaker pattern: si Deepgram o OpenAI están degradados, los jobs se encolan y reintentan sin perder llamadas.
    • 
          

        

        

          
🔒 Seguridad

          
    
            
  • Azure Key Vault: todas las API keys y connection strings. Acceso mediante Managed Identity — sin secretos en variables de entorno ni código.
    • 
            
  • Azure AD + RBAC: cada servicio tiene una identidad gestionada con permisos mínimos. Un Container App solo puede leer el Blob Storage de entrada.
    • 
            
  • Private Endpoints: Blob Storage, PostgreSQL y Cosmos DB sin acceso público. Solo accesibles desde dentro de la VNet.
    • 
            
  • Encryption at rest + in transit: AES-256 en todos los servicios Azure. TLS 1.2+ obligatorio para toda comunicación.
    • 
            
  • PII Masking: antes de enviar transcripciones a GPT-4o, se aplica un paso de anonimización (números de cuenta, nombres) usando Azure AI Language.
    • 
            
  • Audit logs: Azure Monitor + Application Insights registra cada procesamiento con trazabilidad completa para compliance GDPR/PCI-DSS.
    • 
          

        

      

    


    

      📋
      Este es un diseño conceptual — no se implementa código adicional. El objetivo es demostrar la capacidad de diseñar sistemas escalables y justificar decisiones técnicas en un contexto real de call center a escala empresarial.
