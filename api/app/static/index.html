A deep-dive into Deepgram's real-time transcription mode â what it is, how it works, and how the system design would change.

    


    
    

      
What We Built vs What Live Streaming Is


      

        

          The current POST /stt endpoint uses Deepgram's
          pre-recorded mode: you upload a complete audio file,
          wait for it to be fully processed, and receive the entire transcript in one response.
          This is perfect for processing existing recordings.
        

        

          Deepgram's live streaming mode works completely differently:
          instead of sending a finished file, you open a persistent two-way connection and send
          audio in small chunks as it is being recorded â in real time. Deepgram sends back partial
          transcripts within milliseconds as it hears each word, before you've even finished speaking the sentence.
        

      


      

        
          

            Dimension
            Pre-recorded (current)
            Live Streaming
          

        
        
          

            Protocol
            HTTP POST â one request, one response
            WebSocket â persistent two-way connection that stays open
          

          

            Audio input
            Complete file uploaded all at once
            Raw audio chunks sent continuously, 20â250ms at a time
          

          

            When you get results
            After the entire file is processed (seconds to minutes)
            Within ~300ms of each word being spoken â mid-sentence
          

          

            Result type
            One final, complete transcript
            A stream of interim results (partial) followed by final results per utterance
          

          

            Use case
            Podcast transcription, call recording analysis, meeting notes
            Live captioning, voice commands, real-time translation, voice-driven UI
          

          

            Connection lifecycle
            Opens â sends file â closes
            Opens â streams audio chunks â receives words â stays open â closes when done
          

          

            Latency
            Equal to audio duration + processing time
            ~300ms from spoken word to transcript (near real-time)
          

        
      

    


    
    

      
How Deepgram Live Streaming Works â Step by Step

      

        

          

            

            
1. Client opens a WebSocket connection

            

              Instead of a POST request, the client connects to
              wss://api.deepgram.com/v1/listen
              with model parameters in the URL query string (model, language, diarize, etc.).
              The connection stays open for the entire duration of the session.
            

          

          

            

            
2. Client streams raw audio bytes

            

              The microphone (or audio source) is read in small binary chunks â typically 20 to 250 milliseconds of audio at a time.
              Each chunk is sent over the WebSocket as a binary message the moment it is captured.
              No buffering, no waiting for the sentence to finish.
            

          

          

            

            
3. Deepgram sends back interim transcripts

            

              After receiving each chunk, Deepgram responds with a JSON message containing
              is_final: false â a best-guess transcript of what it heard so far.
              These are fast but may change as more context arrives. You can use these to show
              "live captions" that update as the person speaks.
            

          

          

            

            
4. Deepgram sends final utterances

            

              When Deepgram detects a pause (end of sentence or thought), it sends a message with
              is_final: true â the committed, corrected transcript for that utterance.
              This is what you save to your database or display as confirmed captions.
            

          

          

            

            
5. Client sends a CloseStream message

            

              When the user stops speaking or the session ends, the client sends a special
              {"type":"CloseStream"}
              JSON message. Deepgram flushes any remaining audio, sends the last final transcript,
              and closes the connection cleanly.
            

          

        


        

          ðĄ
          A WebSocket is different from HTTP. HTTP is like sending a letter and waiting for a reply â one exchange and done. A WebSocket is like a phone call â the line stays open and both sides can send messages to each other at any time, in any order, for as long as they want.
        

      

    


    
    

      
What Would Change in the System Design


      

        
â Remove â no longer needed

        
The file upload (multipart/form-data): There is no file to upload. Audio arrives as a stream of small binary packets, not as a complete file. The HTTP POST endpoint and UploadFile handler would be replaced entirely.

      


      

        
â Add â new infrastructure required

        
WebSocket endpoint in FastAPI: FastAPI supports WebSockets natively with @app.websocket("/stt/live"). The handler would accept the client's WebSocket connection, open a second WebSocket to Deepgram, and act as a proxy â forwarding audio chunks from the client to Deepgram and forwarding transcript messages from Deepgram back to the client, simultaneously.

      


      

        
â Add â concurrent I/O handling

        
Async bidirectional message routing: The server needs to listen on two WebSocket connections at the same time â one from the browser and one to Deepgram. This requires two concurrent async tasks running in parallel using asyncio.gather() or asyncio.create_task(). If either connection drops, both must be closed cleanly.

      


      

        
ð Modify â different database tracking

        
Database logging strategy: With pre-recorded, you log one row per request. With live streaming, a session can last minutes. The tracking approach changes: you either log one row per final utterance received (many small rows per session), or accumulate the full session transcript in memory and write one row when the session ends. The latency field changes meaning too â it becomes the time from a spoken word to receiving the final transcript for that utterance, not end-to-end request time.

      


      

        
ð Modify â authentication approach

        
API Key in WebSocket: HTTP headers cannot be set on WebSocket connections from a browser. The X-API-Key header would instead be passed as a URL query parameter when opening the connection (wss://host/stt/live?api_key=...), or in the first message sent after the connection opens. The server validates it immediately and closes the connection if it is missing or wrong.

      


      

        
ð Modify â frontend interaction

        
Browser microphone access: The frontend would use the MediaRecorder API to access the microphone, capture audio in chunks, and send each chunk over a WebSocket. The transcript results would appear word-by-word on screen in real time â the UI would update continuously rather than waiting for a single final result to display.

      

    


    
    

      

        
Summary â Pre-recorded vs Live at a Glance

        

          

            
Pre-recorded (implemented)

            
    
              
  • Simple HTTP POST â easy to build and test
    • 
              
  • Works on any audio file regardless of when it was recorded
    • 
              
  • One response contains the complete, final transcript
    • 
              
  • Latency = full audio duration + processing time
    • 
              
  • Best for: archives, recordings, uploads
    • 
            

          

          

            
Live Streaming (reflection)

            
    
              
  • WebSocket â more complex, requires bidirectional handling
    • 
              
  • Only works with live microphone or real-time audio source
    • 
              
  • Words appear on screen as they are spoken (~300ms delay)
    • 
              
  • Latency = milliseconds per word regardless of session length
    • 
              
  • Best for: captions, voice UI, real-time translation
    • 
            

          

        

        

          ðŊ
          Both modes use the same Deepgram Nova-2 model and produce the same quality transcript. The difference is entirely about when you need the results â after the fact (pre-recorded) or as it happens (live streaming).
