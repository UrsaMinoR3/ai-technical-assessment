🔴 Live Streaming STT

      
A deep-dive into Deepgram's real-time transcription mode — what it is, how it works, and how the system design would change.

    


    
    

      
What We Built vs What Live Streaming Is


      

        

          The current POST /stt endpoint uses Deepgram's
          pre-recorded mode: you upload a complete audio file,
          wait for it to be fully processed, and receive the entire transcript in one response.
          This is perfect for processing existing recordings.
        

        

          Deepgram's live streaming mode works completely differently:
          instead of sending a finished file, you open a persistent two-way connection and send
          audio in small chunks as it is being recorded — in real time. Deepgram sends back partial
          transcripts within milliseconds as it hears each word, before you've even finished speaking the sentence.
        

      


      

        
          

            Dimension
            Pre-recorded (current)
            Live Streaming
          

        
        
          

            Protocol
            HTTP POST — one request, one response
            WebSocket — persistent two-way connection that stays open
          

          

            Audio input
            Complete file uploaded all at once
            Raw audio chunks sent continuously, 20–250ms at a time
          

          

            When you get results
            After the entire file is processed (seconds to minutes)
            Within ~300ms of each word being spoken — mid-sentence
          

          

            Result type
            One final, complete transcript
            A stream of interim results (partial) followed by final results per utterance
          

          

            Use case
            Podcast transcription, call recording analysis, meeting notes
            Live captioning, voice commands, real-time translation, voice-driven UI
          

          

            Connection lifecycle
            Opens → sends file → closes
            Opens → streams audio chunks → receives words → stays open → closes when done
          

          

            Latency
            Equal to audio duration + processing time
            ~300ms from spoken word to transcript (near real-time)
          

        
      

    


    
    

      
How Deepgram Live Streaming Works — Step by Step

      

        

          

            

            
1. Client opens a WebSocket connection

            

              Instead of a POST request, the client connects to
              wss://api.deepgram.com/v1/listen
              with model parameters in the URL query string (model, language, diarize, etc.).
              The connection stays open for the entire duration of the session.
            

          

          

            

            
2. Client streams raw audio bytes

            

              The microphone (or audio source) is read in small binary chunks — typically 20 to 250 milliseconds of audio at a time.
              Each chunk is sent over the WebSocket as a binary message the moment it is captured.
              No buffering, no waiting for the sentence to finish.
            

          

          

            

            
3. Deepgram sends back interim transcripts

            

              After receiving each chunk, Deepgram responds with a JSON message containing
              is_final: false — a best-guess transcript of what it heard so far.
              These are fast but may change as more context arrives. You can use these to show
              "live captions" that update as the person speaks.
            

          

          

            

            
4. Deepgram sends final utterances

            

              When Deepgram detects a pause (end of sentence or thought), it sends a message with
              is_final: true — the committed, corrected transcript for that utterance.
              This is what you save to your database or display as confirmed captions.
            

          

          

            

            
5. Client sends a CloseStream message

            

              When the user stops speaking or the session ends, the client sends a special
              {"type":"CloseStream"}
              JSON message. Deepgram flushes any remaining audio, sends the last final transcript,
              and closes the connection cleanly.
            

          

        


        

          💡
          A WebSocket is different from HTTP. HTTP is like sending a letter and waiting for a reply — one exchange and done. A WebSocket is like a phone call — the line stays open and both sides can send messages to each other at any time, in any order, for as long as they want.
        

      

    


    
    

      
What Would Change in the System Design


      

        
❌ Remove — no longer needed

        
The file upload (multipart/form-data): There is no file to upload. Audio arrives as a stream of small binary packets, not as a complete file. The HTTP POST endpoint and UploadFile handler would be replaced entirely.

      


      

        
✅ Add — new infrastructure required

        
WebSocket endpoint in FastAPI: FastAPI supports WebSockets natively with @app.websocket("/stt/live"). The handler would accept the client's WebSocket connection, open a second WebSocket to Deepgram, and act as a proxy — forwarding audio chunks from the client to Deepgram and forwarding transcript messages from Deepgram back to the client, simultaneously.

      


      

        
✅ Add — concurrent I/O handling

        
Async bidirectional message routing: The server needs to listen on two WebSocket connections at the same time — one from the browser and one to Deepgram. This requires two concurrent async tasks running in parallel using asyncio.gather() or asyncio.create_task(). If either connection drops, both must be closed cleanly.

      


      

        
🔄 Modify — different database tracking

        
Database logging strategy: With pre-recorded, you log one row per request. With live streaming, a session can last minutes. The tracking approach changes: you either log one row per final utterance received (many small rows per session), or accumulate the full session transcript in memory and write one row when the session ends. The latency field changes meaning too — it becomes the time from a spoken word to receiving the final transcript for that utterance, not end-to-end request time.

      


      

        
🔄 Modify — authentication approach

        
API Key in WebSocket: HTTP headers cannot be set on WebSocket connections from a browser. The X-API-Key header would instead be passed as a URL query parameter when opening the connection (wss://host/stt/live?api_key=...), or in the first message sent after the connection opens. The server validates it immediately and closes the connection if it is missing or wrong.

      


      

        
🔄 Modify — frontend interaction

        
Browser microphone access: The frontend would use the MediaRecorder API to access the microphone, capture audio in chunks, and send each chunk over a WebSocket. The transcript results would appear word-by-word on screen in real time — the UI would update continuously rather than waiting for a single final result to display.

      

    


    
    

      

        
Summary — Pre-recorded vs Live at a Glance

        

          

            
Pre-recorded (implemented)

            
    
              
  • Simple HTTP POST — easy to build and test
    • 
              
  • Works on any audio file regardless of when it was recorded
    • 
              
  • One response contains the complete, final transcript
    • 
              
  • Latency = full audio duration + processing time
    • 
              
  • Best for: archives, recordings, uploads
    • 
            

          

          

            
Live Streaming (reflection)

            
    
              
  • WebSocket — more complex, requires bidirectional handling
    • 
              
  • Only works with live microphone or real-time audio source
    • 
              
  • Words appear on screen as they are spoken (~300ms delay)
    • 
              
  • Latency = milliseconds per word regardless of session length
    • 
              
  • Best for: captions, voice UI, real-time translation
    • 
            

          

        

        

          🎯
          Both modes use the same Deepgram Nova-2 model and produce the same quality transcript. The difference is entirely about when you need the results — after the fact (pre-recorded) or as it happens (live streaming).

📚 Before You Start — Core Concepts

      

        If this is your first time working with an API, read this section first. Everything in this documentation will make much more sense once you understand these five ideas.
      

      


        

          
🍽️

          
What is an API?

          

            Think of a restaurant. You (the customer) don't go into the kitchen yourself — you talk to a waiter who takes your order to the kitchen and brings back your food. An API is that waiter. Your app makes a request, the API takes it to the right service, and brings back the result. You never touch the "kitchen" (Deepgram, OpenAI) directly.
          

        


        

          
📬

          
What is HTTP?

          

            HTTP is how computers send messages over the internet. Every request has a method (what you want to do) and a URL (where to send it).
            


            POST = "I'm sending you something, please process it" (like uploading a file).
            
GET = "Give me some information" (like fetching logs).
          

        


        

          
📦

          
What is JSON?

          

            JSON is a standard text format for organizing data — like a very organized shopping list. Fields have names and values, separated by colons, wrapped in curly braces:
            


            { "name": "David", "age": 30 }
            


            Every API response in this system is JSON — a structured object you can read or process with code.
          

        


        

          
🏷️

          
What is a Header?

          

            Headers are labels that travel with every request, invisible to the user but read by the server. Like writing your name and address on an envelope before mailing it — the package has the content, the envelope has the metadata.
            


            This API uses a header called X-API-Key to verify your identity on every request.
          

        


        

          
📎

          
Form Upload vs JSON Body

          

            Two ways to send data to an API:
            


            JSON body — plain text formatted as JSON. Used when sending simple values like text or numbers.
            


            Form/multipart — used when attaching actual files (like audio or images). The same way a website form works when you upload a profile picture.
          

        


        

          
🔢

          
What are Status Codes?

          

            Every API response includes a 3-digit number telling you if it worked:
            


            200 = Success — everything worked.
            
401 = Unauthorized — wrong or missing API key.
            
422 = Bad request — you sent the wrong data format.
            
500 = Server error — something broke on our end.
          

        


      

    


    
    

      
🔐 Authentication — How to Prove You Have Access

      

        Every request to this API (except /health) requires an API Key sent as a header.


        An API Key is a secret string that acts as your password. You include it on every single request so the server knows the request is legitimate and not from a random person on the internet.


        Header name: X-API-Key  
        Value: blackpool-architect-key-2024


        If you forget to include it, or send the wrong value, the API immediately returns 401 Unauthorized and does not process your request at all. On this website, the key is already pre-filled in the 🔑 field at the top — you never need to type it manually here.

Option C — Use Python (for developers)

        

          Install the requests library first: pip install requests. Then use the examples below. This is the most common way to integrate this API into a real application.
        

        

          
Python — all 3 endpoints

          
import requests

BASE = "http://localhost:8000"
HEADERS = {"X-API-Key": "blackpool-architect-key-2024"}

# ── Speech-to-Text ──────────────────────────────────────────────
with open("recording.mp3", "rb") as audio_file:
    response = requests.post(
        f"{BASE}/stt",
        headers=HEADERS,
        files={"audio": audio_file}   # file upload, not JSON
    )
data = response.json()
print("Transcript:", data["transcript"])
print("Language:", data["language"])
print("Duration:", data["duration"], "seconds")

# ── Text-to-Speech ──────────────────────────────────────────────
response = requests.post(
    f"{BASE}/tts",
    headers=HEADERS,
    json={                            # JSON body
        "text": "Hello! This is a test of the text to speech system.",
        "voice": "aura-2-asteria-en"
    }
)
with open("output.mp3", "wb") as f:  # save binary MP3 to disk
    f.write(response.content)
print("Audio saved to output.mp3")

# ── Document Analysis ───────────────────────────────────────────
with open("document.jpg", "rb") as image_file:
    response = requests.post(
        f"{BASE}/idp/analyze",
        headers=HEADERS,
        files={"image": image_file}   # file upload, not JSON
    )
data = response.json()
print("Document type:", data["document_type"])
print("Confidence:", data["confidence"])
for field, value in data["extracted_fields"].items():
    print(f"  {field}: {value}")

        

        

          💡
          Notice the difference: STT and IDP use files= because they receive actual file uploads. TTS uses json= because it receives plain text data. Mixing these up is the most common beginner mistake.
        

      

    


    
    

      
Endpoint 1 — Speech-to-Text (STT)

      

        

          POST
          /stt
          Powered by Deepgram Nova-2
        

        


          
What it does: You upload an audio file (MP3 or WAV). The server sends it to Deepgram's Nova-2 AI speech recognition model. Nova-2 listens to the entire recording, converts every spoken word to text, detects what language is being spoken automatically, measures how long the audio lasts, and figures out which voice belongs to which speaker when there are multiple people talking. You get back all of this as structured JSON.


          
Real-world example: A company records all their customer support calls. Every night, they run those recordings through this endpoint to get transcripts, then search the transcripts for keywords like "cancel" or "refund" to identify at-risk customers — without anyone listening to the calls manually.

→

            
🔐 API Key check

            
→

            
☁️ Deepgram Nova-2

            
→

            
📝 Transcript + speakers

            
→

            
🗄️ Logged to DB

          


          

            

              
What you send

              
    
                
  • audio — the file itself (MP3 or WAV, max 25 MB). This is sent as a file attachment, exactly like attaching a file to an email — not as JSON text.
    • 
                
  • X-API-Key header — your key. Without this, the request is rejected before the file is even looked at.
    • 
              

            

            

              
What you get back

              
    
                
  • transcript — all the spoken words, in order, as a single text string.
    • 
                
  • language — the language code Deepgram detected: en (English), es (Spanish), de (German), etc.
    • 
                
  • duration — how many seconds long the audio was.
    • 
                
  • speakers — a list of segments. Each segment has a speaker number, start/end time in seconds, and the exact words that person said.
    • 
              

            

          


          

            
Example response — two speakers in a Spanish conversation

            
{
  "transcript": "Hola, ¿cómo estás hoy? Estoy bien, gracias.",
  "language": "es",
  "duration": 5.20,
  "speakers": [
    { "speaker": 0, "start": 0.0,  "end": 2.8,  "text": "Hola, ¿cómo estás hoy?" },
    { "speaker": 1, "start": 3.1,  "end": 5.2,  "text": "Estoy bien, gracias." }
  ]
}

          


          

            ⚠️
            Speaker 0 and Speaker 1 are just labels — Deepgram doesn't know their names, only that they are different voices. In the example above, speaker 0 asked a question and speaker 1 replied.
          


          

            💡
            Deepgram charges per second of audio. The duration field tells you exactly how many seconds were billed. This is automatically saved to the database under external_usage → deepgram_duration_s, so you can always audit costs later.

Endpoint 2 — Text-to-Speech (TTS)

      

        

          POST
          /tts
          Powered by Deepgram Aura-2
        

        


          
What it does: You send a piece of text (in any of 7 supported languages) and choose an AI voice. The server sends it to Deepgram's Aura-2 text-to-speech model, which reads your text aloud in that voice, and returns the resulting audio as an MP3 file you can play directly or download.


          
Real-world example: A navigation app reads turn-by-turn directions aloud. An e-commerce site reads product descriptions for visually impaired users. A podcast platform converts written articles into audio episodes automatically. A bank IVR system speaks account balances to customers over the phone.

What you send (JSON body)

              
    
                
  • text — the words to be spoken. Can be 1 character to 2,000 characters. Supports any language in the Aura-2 voice list.
    • 
                
  • voice — (optional) the voice ID to use. Example: aura-2-asteria-en. If you omit this field, the API uses Asteria (English female) as the default. The full list of 80+ voices covers English, Spanish, German, Dutch, Italian, French, and Japanese.
    • 
              

            

            

              
What you get back

              
    
                
  • audio/mpeg binary file — raw MP3 audio bytes. This is NOT JSON text — it is binary audio data that your browser can play directly or that code can save to a .mp3 file.
    • 
                
  • On this website, the audio player appears automatically and starts playing the result immediately after the request finishes.
    •

{
  "text": "Bienvenido a Blackpool. ¿En qué puedo ayudarte hoy?",

// Response: binary MP3 data (not JSON)

This prevents selecting an English voice and making it try to read Spanish text.
          


          

            💡
            Deepgram Aura-2 charges per character of text synthesized (including spaces). The exact character count is tracked in every database log entry under external_usage → chars_synthesized.

Endpoint 3 — Intelligent Document Processing (IDP)

      

        

          POST
          /idp/analyze
          Powered by Azure OpenAI GPT-4o Vision
        

        


          
What it does: You upload a photo or scan of any document — an ID card, passport, driver's license, invoice, receipt, or even a handwritten form. The server sends the image to OpenAI's GPT-4o Vision model, which looks at the image like a human reader would, identifies what type of document it is, reads every visible text field, and returns all the extracted data organized as clean JSON. It also calculates where on the image it found each field (bounding boxes), which is used to draw colored annotation boxes on the document image in this website.


          
Real-world example: A bank uses this to process ID verification during new account opening — a customer photographs their ID, this endpoint extracts the name, date of birth, ID number, and expiry date automatically, and the bank's system pre-fills the application form without anyone typing. What used to take 3 minutes of manual data entry takes 2 seconds.


          

            
🖼️ Document photo

            
→

            
🔐 API Key check

            
→

            
☁️ GPT-4o Vision

            
→

            
📋 Fields + bounding boxes

            
→

            
🗄️ Logged to DB

          


          

            

              
What you send

              
    
                
  • image — the document photo (JPG or PNG only, max 10 MB). Must be a real file upload — not a URL, not base64 text. Just the file itself.
    • 
                
  • X-API-Key header — your key.
    • 
                
  • The image quality matters: blurry, dark, or heavily cropped images will reduce confidence and may miss fields. A clear, flat-angle photo in good lighting gives the best results.
    • 
              

            

            

              
What you get back

              
    
                
  • document_type — what the model identified the document as (e.g. "Passport", "National ID Card", "Driver's License", "Invoice").
    • 
                
  • extracted_fields — every readable field as a name/value pair. The field names adapt to the document type (a passport has "passport_number"; an invoice has "total_amount").
    • 
                
  • annotations — for each field, the approximate location on the image as four numbers [x1, y1, x2, y2] ranging 0.0 to 1.0. These are fractions of the image size, not pixel coordinates. Used to draw the colored boxes on screen.
    • 
                
  • confidence — high (model is certain), medium (some fields may be uncertain), or low (blurry image or unusual document — verify manually).
    • 
              

            

          


          

            
Example response — Colombian passport

            
{
  "document_type": "Passport",
  "confidence": "high",
  "extracted_fields": {
    "surname": "RODRIGUEZ",
    "given_names": "CLAUDIA MARCELA",
    "nationality": "COLOMBIANA",
    "date_of_birth": "20 MAY 1982",
    "passport_number": "CB007001",
    "expiry_date": "15 NOV 2030"
  },
  "annotations": {
    "surname":         { "value": "RODRIGUEZ",      "bbox": [0.1, 0.38, 0.6, 0.45] },
    "given_names":     { "value": "CLAUDIA MARCELA","bbox": [0.1, 0.46, 0.75,0.53] },
    "passport_number": { "value": "CB007001",        "bbox": [0.6, 0.12, 0.95,0.20] }
  }
}

          


          

            🎯
            What are bounding boxes? Each bbox is [x1, y1, x2, y2] — the left, top, right, and bottom edges of a rectangle around the field on the image. All values are between 0.0 (left/top edge) and 1.0 (right/bottom edge). On this website, these coordinates are used to draw the colored annotation boxes you see overlaid on the document image when you zoom in.
          


          

            💡
            OpenAI GPT-4o charges per token — roughly, pieces of words. A typical document analysis uses 400–600 prompt tokens (the image + instructions sent to the model) and 200–400 completion tokens (the JSON it writes back). Token counts are tracked under external_usage in every database log entry.

Endpoint 4 — Request Logs

      

        

          GET
          /logs
          PostgreSQL — full request history
        

        


          
What it does: Returns the full history of every API call made through this system, read directly from the PostgreSQL database. Each record answers the questions: when did it happen, which feature was used, what data was sent in, what came out, how long it took, and how much of each external paid service (Deepgram, OpenAI) was consumed.


          
Why does this exist? Every system that calls paid external AI services needs to track its own usage. Without this, you couldn't answer questions like: "How many seconds of audio did we transcribe this month?", "Which request took 12 seconds to respond?", or "Did the 3pm request yesterday fail or succeed?"


          

            

              
What you send

              
    
                
  • X-API-Key header — required. Even viewing logs requires authentication.
    • 
                
  • ?limit=N — optional URL parameter to control how many records come back. Example: /logs?limit=10 returns the 10 most recent requests. Default: 100.
    • 
              

            

            

              
Fields returned per record — what they mean

              
    
                
  • id — unique row number in the database.
    • 
                
  • timestamp — exact date and time the request happened (UTC timezone).
    • 
                
  • endpoint — which feature was called: /stt, /tts, or /idp/analyze.
    • 
                
  • input_data — a summary of what was sent in (filename for audio/image, text snippet for TTS).
    • 
                
  • output_ref — a summary of what came back (transcript preview, audio size, field count).
    • 
                
  • external_usage — exactly how much of the paid external service was used: seconds of audio for STT, characters for TTS, tokens for IDP.
    • 
                
  • latency_ms — how many milliseconds the full request took, from the moment it arrived to the moment the response was sent.
    • 
                
  • http_status — 200 means success. 422 means bad input. 500 means something went wrong internally.
    • 
              

            

          


          

            
Example single log record

            
{
  "id": 42,
  "timestamp": "2024-12-15T14:32:07.123456",
  "endpoint": "/stt",
  "input_data": "meeting_recording.mp3",
  "output_ref": "transcript_preview: 'Good afternoon everyone, let's...'",
  "external_usage": { "deepgram_duration_s": 187.4 },
  "latency_ms": 3240.5,
  "http_status": 200
}

          


          

            💡
            You can also view this data visually on the Database panel in the sidebar — it shows a summary card plus a full sortable table with all the same fields, without needing to read raw JSON.
          

        

      

    


    
    

      
Endpoint 5 — Health Check

      

        

          GET
          /health
          No authentication required
        

        


          
What it does: A minimal "is the server online?" check. It returns a tiny JSON object with two fields: the current status ("ok") and the API version. That's it — no database queries, no AI calls, no authentication.


          
Why does this exist? Every professionally deployed service needs a health endpoint. It's used by: deployment pipelines (to verify a new version started successfully), load balancers (to stop routing traffic to a crashed instance), uptime monitors (to alert on-call staff if the service goes down), and container orchestrators like Kubernetes (to decide whether to restart a pod).


          
Why no API key? Health checks need to work even if the authentication system is broken. An uptime monitor needs to know whether the server is reachable at all — requiring a key would prevent that check from working if key validation itself has a bug.


          

            
Request

            
GET http://localhost:8000/health
# No headers required

          

          

            
Response

            
{ "status": "ok", "version": "1.0.0" }

          

        

      

    


    
    

      
⚠️ Common Errors and What They Mean

      

        

          
            

              Code
              What it means
              How to fix it
            

          
          
            

              200 OK
              The request worked. The response body contains your result.
              Nothing to fix — read the response JSON or play the audio.
            

            

              401
              Unauthorized — the X-API-Key header is missing or the key value is wrong.
              Add the header X-API-Key: blackpool-architect-key-2024 to your request. On this website, check the 🔑 field at the top.
            

            

              404
              Not Found — you requested a URL path that doesn't exist.
              Check the endpoint path. Common mistake: typing /transcribe instead of /stt, or missing a slash.
            

            

              405
              Method Not Allowed — you used GET on an endpoint that only accepts POST, or vice versa.
              STT, TTS, and IDP all require POST. Health and Logs require GET. Use the correct HTTP method.
            

            

              422
              Unprocessable Entity — the server understood your request but the data you sent is in the wrong format.
              Common causes: sending a JSON body when a file upload is expected (or vice versa), sending a PNG when only JPG is accepted, or omitting a required field. Check the "What you send" section for the endpoint you're calling.
            

            

              500
              Internal Server Error — something went wrong inside the server after it accepted your request.
              Could be a bug in the code, a timeout from Deepgram or OpenAI, or a database connection issue. Check the server logs. The request is still recorded in the database with http_status: 500.
            

            

              503
              Service Unavailable — the server is overloaded or an external dependency (Deepgram, OpenAI) is down.
              Wait a few seconds and retry. If the error persists, check the health endpoint first (GET /health). If that also fails, the server itself is down.
            

          
        

      


      

        🔍
        Every response from this API includes the status code in the HTTP headers. When using cURL, add -i to your command to see the full headers including the status code. Example: curl -i http://localhost:8000/health
      

    


  


  
  

    

      
☁️ Architecture Point 2

      
Diseño conceptual de arquitectura cloud — Caso A: Análisis masivo de llamadas en call center.

    


    
    

      
    


    
    

      
1. Descripción del Caso de Uso y Retos Técnicos

      

        

          Un call center de gran escala puede manejar entre 50,000 y 500,000 llamadas diarias.
          Cada llamada genera un archivo de audio que, en el modelo tradicional, requiere revisión humana para extraer insights:
          ¿el agente fue empático? ¿el cliente quedó satisfecho? ¿de qué tema trató la llamada?
          El objetivo de esta arquitectura es procesar ese volumen de manera completamente automática,
          en la nube, produciendo transcripciones, análisis de sentimiento, detección de temas y reportes por agente y por campaña,
          sin intervención humana en el pipeline.
        

        
RETOS TÉCNICOS PRINCIPALES

        

          

            
📦 Volumen e ingesta

            
    
              
  • Cientos de miles de archivos de audio diarios llegando de forma continua
    • 
              
  • Distintos formatos (WAV, MP3, OGG) y distintas duraciones (2–45 min)
    • 
              
  • Picos de carga en horarios de mayor actividad del call center
    • 
            

          

          

            
⚡ Latencia y tiempo real

            
    
              
  • Los supervisores necesitan alertas de llamadas críticas en <5 minutos
    • 
              
  • Transcripción y análisis deben terminar antes del siguiente turno
    • 
              
  • Soporte para modo live (streaming) para monitoreo en tiempo real
    • 
            

          

          

            
🌍 Multilingüismo

            
    
              
  • Call centers atienden en múltiples idiomas según región
    • 
              
  • El modelo STT debe detectar idioma automáticamente
    • 
              
  • Los análisis de sentimiento deben ser culturalmente precisos
    • 
            

          

          

            
🔒 Privacidad y compliance

            
    
              
  • Datos de clientes altamente sensibles (PII, datos financieros)
    • 
              
  • Regulaciones como GDPR, PCI-DSS según país
    • 
              
  • Trazabilidad y auditoría completa de cada procesamiento
    •

3. Flujo de Datos — Entrada, Procesamiento y Salida

      

        

          

            

            
① Entrada — Grabación llega al sistema

            

              El PBX o sistema de grabación del call center sube el archivo WAV/MP3 a Azure Blob Storage (RAW)
              vía API REST o SFTP. Cada archivo va etiquetado con metadatos: ID de agente, ID de campaña, timestamp de inicio,
              número de cliente (anonimizado para compliance). En el momento del upload, Azure Event Grid
              dispara un evento que crea un mensaje en Azure Service Bus sin ninguna acción manual.
            

          

          

            

            
② Procesamiento STT — Transcripción con Deepgram

            

              Azure Container Apps consume el mensaje de la cola, descarga el audio de Blob Storage
              y lo envía a Deepgram Nova-2. El modelo devuelve la transcripción completa con
              diarización (quién habló en qué momento), idioma detectado y marcas de tiempo por palabra.
              Para llamadas largas (>30 min), el audio se divide en segmentos de 5 minutos para paralelizar
              el procesamiento y reducir la latencia total.
            

          

          

            

            
③ Análisis NLU — Azure OpenAI GPT-4o

            

              La transcripción es enviada a Azure OpenAI (GPT-4o) con un prompt estructurado que extrae:
              sentimiento general (positivo / neutro / negativo / escalar 0–100),
              temas detectados (billing, soporte técnico, cancelación, queja, etc.),
              entidades clave (productos mencionados, números de cuenta — con enmascaramiento PII),
              resumen ejecutivo de la llamada (<3 oraciones),
              y score de calidad del agente (adherencia a protocolo, empatía, resolución).
              Los resultados se estructuran como JSON para escritura directa en base de datos.
            

          

          

            

            
④ Persistencia — Múltiples destinos en paralelo

            

              El orquestador escribe de forma paralela en tres destinos:
              Azure PostgreSQL recibe la fila de metadatos y scores (escritura rápida, consultable con SQL estándar).
              Cosmos DB recibe el documento JSON completo con la transcripción y el resumen.
              ADLS Gen2 archiva todos los datos crudos y procesados para compliance y re-entrenamiento futuro.
            

          

          

            

            
⑤ Salida — Reportes y alertas

            

              Azure Synapse Analytics corre queries programadas (cada hora o bajo demanda) que agregan los datos
              de PostgreSQL y el Data Lake para producir métricas consolidadas.
              Power BI se conecta a Synapse y presenta dashboards en tiempo cuasi-real: ranking de agentes
              por score de calidad, distribución de sentimiento por campaña, temas más frecuentes, evolución temporal.
              Si el análisis detecta sentimiento muy negativo durante el procesamiento, Azure Monitor
              dispara una alerta al supervisor en menos de 5 minutos tras la finalización de la llamada.
            

          

        

      

    


    
    

      
4. Justificación de los Servicios Cloud Elegidos

      

        

          
            

              Servicio
              Por qué este servicio
              Por qué no otra alternativa
            

          
          
            

              Deepgram Nova-2
              Mejor WER (Word Error Rate) en inglés y español para audio de call center con ruido. Diarización nativa, detección automática de idioma y marcas de tiempo por palabra en una sola llamada API.
              Azure Speech-to-Text tiene menor precisión en acentos regionales. OpenAI Whisper no ofrece diarización nativa ni streaming en tiempo real.
            

            

              Azure OpenAI GPT-4o
              Modelo líder en comprensión de texto complejo. Permite análisis multi-tarea en un solo prompt (sentimiento + temas + entidades + resumen), reduciendo el número de llamadas API.
              Modelos más pequeños (GPT-3.5) son menos precisos en análisis de matices de sentimiento. Modelos open-source requieren infraestructura propia y ajuste fino costoso.
            

            

              Azure Blob Storage
              Almacenamiento de objetos de bajo costo con 11 nueves de durabilidad. Integración nativa con Event Grid para triggers automáticos sin polling.
              ADLS Gen2 es superior para analytics pero más costoso como landing zone. Blob Storage es óptimo para la ingesta inicial de archivos binarios.
            

            

              Azure Service Bus
              Garantiza entrega al menos una vez, reintentos automáticos, dead-letter queue para mensajes fallidos y desacoplamiento total entre ingesta y procesamiento.
              Azure Event Hubs está orientado a streaming de alta frecuencia y no garantiza orden ni exactly-once delivery. Para jobs discretos de procesamiento de archivos, Service Bus es más apropiado.
            

            

              Azure Container Apps
              Escala automáticamente a 0 cuando no hay mensajes en la cola (KEDA) y a cientos de réplicas en momentos pico — sin gestionar infraestructura de Kubernetes manualmente.
              Azure Functions tiene límite de ejecución de 10 min (máx 60), insuficiente para llamadas largas. AKS requiere gestión activa del cluster. Container Apps es el punto medio ideal.
            

            

              Azure PostgreSQL Flexible
              Base de datos relacional gestionada con alta disponibilidad, backups automáticos y soporte nativo para JSON. Compatible con el ORM (SQLAlchemy) ya usado en este proyecto.
              Cosmos DB no es óptimo para queries SQL relacionales con JOINs y GROUP BY que producen los reportes de agentes. PostgreSQL es mejor para datos estructurados y métricas.
            

            

              Azure Cosmos DB
              Latencia de lectura <10ms para servir transcripciones completas bajo demanda. Modelo de datos flexible (JSON) ideal para documentos de longitud variable sin esquema fijo.
              PostgreSQL no es eficiente almacenando textos de transcripción de 5,000+ palabras como campos JSONB si se consultan frecuentemente por fragmento de texto.
            

            

              Azure Synapse Analytics
              Permite consultar el Data Lake (ADLS) con SQL serverless sin mover datos. Escala elásticamente para queries históricas pesadas sin costo fijo de infraestructura.
              Azure SQL Warehouse (dedicado) tiene costo fijo aunque no se use. Synapse serverless paga por TB leído — ideal para analytics episódicos sobre histórico grande.
            

            

              Power BI
              Integración nativa con Azure Synapse y PostgreSQL. Dashboards interactivos sin necesidad de código. Familiar para supervisores y equipos de gestión no técnicos.
              Grafana está más orientado a observabilidad de infraestructura. Tableau requiere licencias más costosas. Power BI Premium está incluido en muchas licencias M365 empresariales.
            

          
        

      

    


    
    

      
5. Escalabilidad, Disponibilidad y Seguridad

      

        

          
📈 Escalabilidad

          
    
            
  • Azure Container Apps + KEDA: escala de 0 a N réplicas basándose en el número de mensajes en Service Bus. 0 costo cuando no hay llamadas.
    • 
            
  • Service Bus Premium con particionamiento: distribuye la carga entre múltiples brokers para alta frecuencia de mensajes.
    • 
            
  • Deepgram concurrencia: soporta miles de transcripciones simultáneas — se paga por minuto de audio, no por instancia.
    • 
            
  • Azure OpenAI rate limits: gestionados con retry + backoff exponencial. Batch API para análisis no urgentes a menor costo.
    • 
            
  • Synapse serverless: escala automáticamente para queries de reportes, sin provisionar capacidad fija.
    • 
            
  • Medallion architecture: Bronze (raw) → Silver (cleaned) → Gold (aggregated). Permite re-procesar solo lo que cambió sin rehacer todo el pipeline.
    • 
          

        

        

          
🔄 Disponibilidad

          
    
            
  • Azure SLA 99.9%+ en todos los servicios gestionados (Blob, PostgreSQL, Cosmos DB, Service Bus).
    • 
            
  • Geo-replication: Blob Storage y Cosmos DB replicados en zona secundaria con failover automático.
    • 
            
  • PostgreSQL Flexible con High Availability: réplica standby en otra zona con failover automático en <60 segundos.
    • 
            
  • Service Bus Dead-Letter Queue: mensajes que fallan 3 veces van a DLQ para revisión, sin pérdida de datos.
    • 
            
  • Container Apps health checks: reinicio automático de instancias no saludables.
    • 
            
  • Circuit breaker pattern: si Deepgram o OpenAI están degradados, los jobs se encolan y reintentan sin perder llamadas.
    • 
          

        

        

          
🔒 Seguridad

          
    
            
  • Azure Key Vault: todas las API keys y connection strings. Acceso mediante Managed Identity — sin secretos en variables de entorno ni código.
    • 
            
  • Azure AD + RBAC: cada servicio tiene una identidad gestionada con permisos mínimos. Un Container App solo puede leer el Blob Storage de entrada.
    • 
            
  • Private Endpoints: Blob Storage, PostgreSQL y Cosmos DB sin acceso público. Solo accesibles desde dentro de la VNet.
    • 
            
  • Encryption at rest + in transit: AES-256 en todos los servicios Azure. TLS 1.2+ obligatorio para toda comunicación.
    • 
            
  • PII Masking: antes de enviar transcripciones a GPT-4o, se aplica un paso de anonimización (números de cuenta, nombres) usando Azure AI Language.
    • 
            
  • Audit logs: Azure Monitor + Application Insights registra cada procesamiento con trazabilidad completa para compliance GDPR/PCI-DSS.
    • 
          

        

      

    


    

      📋
      Este es un diseño conceptual — no se implementa código adicional. El objetivo es demostrar la capacidad de diseñar sistemas escalables y justificar decisiones técnicas en un contexto real de call center a escala empresarial.
